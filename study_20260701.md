markdown# 🛠 Git & GitHub 핵심 요약

## 1. 버전 관리의 필요성
* **버전 관리란?**: 코드의 변경 이력을 체계적으로 기록하고 관리하는 것.
* **필요성**: 변경 이력 추적, 실수 복구, 여러 사람과의 충돌 없는 협업을 위해 필수.
* **Git vs 파일 복사**: "최종_진짜최종" 같은 복사 방식 대신 커밋 단위로 이력을 명확히 기록.
* **Git vs GitHub**: Git은 내 컴퓨터에서 도는 엔진, GitHub는 저장소를 올리는 웹 서비스.

## 2. Git 기본 개념 및 영역
* **Repository**: 프로젝트 전체와 버전 기록이 들어있는 저장소 (로컬/원격).
* **Commit**: 변경 사항을 하나의 기록으로 저장하는 단위 (하나의 버전).
* **HEAD**: 현재 내가 작업 중인 위치를 가리키는 포인터.
* **3대 작업 영역**:
  * **Working Directory**: 파일 수정 및 실제 작업 공간.
  * **Staging Area**: 다음 커밋에 포함할 변경 사항을 미리 모아두는 곳.
  * **Repository**: 최종 커밋이 영구적으로 기록되는 저장소.

## 3. 초기 설정 및 필수 명령어
* **사용자 등록 (전역)**: `git config --global user.name "이름"`, `git config --global user.email "이메일"`
* **저장소 초기화**: `git init` (`.git` 폴더 생성)
* **상태 확인**: `git status` (수정/스테이징 상태 확인)
* **스테이징**: `git add 파일명` 또는 `git add .` (전체)
* **버전 기록**: `git commit -m "커밋메시지"` (명확·간결, 동사로 시작 권장)
* **.gitignore**: 환경 변수(`.env`), 빌드 결과물, 로그 등 관리 제외 파일 설정.

## 4. 브랜치(Branch)와 병합
* **Branch**: 기존 코드와 분리된 독립적인 작업 공간 (안전한 기능 개발).
  * 생성: `git branch 브랜치명`
  * 전환: `git switch 브랜치명`
* **브랜치 관리 전략 예시**: `main`(배포용), `dev`(테스트용), `feature/*`(기능 개발).
* **병합(Merge)**: `git merge 브랜치명`으로 현재 브랜치에 합침.
  * **Fast-forward**: 분기가 없을 때 커밋 포인터만 이동.
  * **3-way merge**: 분기가 있을 때 갈라진 버전을 합쳐 병합 커밋 생성.
* **충돌(Merge Conflict)**: 같은 파일의 같은 부분을 다르게 수정 시 발생 ➡️ 파일 수정 후 다시 커밋.

## 5. 원격 저장소 및 협업
* **동기화**:
  * `git push`: 로컬 커밋을 원격 저장소로 업로드.
  * `git pull`: 원격 저장소 변경 사항을 내려받아 로컬에 반영.
* **Clone vs Fork**:
  * **Clone**: push 권한이 있는 팀 내부 저장소를 그대로 복사.
  * **Fork**: 권한이 없는 타인의 저장소를 내 GitHub 계정으로 복사 (오픈소스 기여).
* **Pull Request (PR)**: 내가 변경한 사항을 원본 저장소에 반영해달라고 요청하는 단계.
* **Code Review**: 버그 예방과 코드 품질을 위해 병합 전 동료들과 확인하는 과정.

## 6. 자주 사용하는 명령어
* **상세 내역 확인**: `git show` (특정 커밋의 변경 내용, 메시지 등 출력)
* **차이점 비교**: `git diff` (영역 간, 커밋 간 차이 분석)
